# Implementation Plan: DB-Backed Integrations, Lead Import, Visits/Demos

**Goals**
- Remove all remaining mock/in-memory data usage so UI only reflects database-backed data.
- Add dynamic lead import (CSV + XLSX) end-to-end.
- Add sales visit + demo scheduling with DB persistence and UI.

**Decisions (based on your guidance: “csv and xlsx; implement the best”)**
- Import formats: CSV and XLSX.
- Required columns: `name`, `company`.
- Optional columns: `email`, `phone`, `source`, `status`, `priority`, `assignedToUserId`, `assignedTo`, `estimatedValue`, `tags`, `notes`, `followUpAt`.
- Duplicate handling: update by email if present, else update by phone; otherwise create new.
- Import errors: continue and report row errors; return summary.
- Schedule types: `VISIT`, `DEMO`.
- Schedule fields: required `type`, `scheduledAt`; optional `location`, `mode (ON_SITE|REMOTE)`, `notes`, `assignedToUserId`.
- Schedule status values: `SCHEDULED`, `COMPLETED`, `CANCELLED`.
- Notifications: none by default (can add WhatsApp later on request).

**Plan Steps**
1. Remove mock/fallback data usage in frontend and backend
   - Remove any in-memory fallback lists in UI (e.g., assignee fallbacks, sample lists) and any sample seeders for CRM/Identity if still enabled.
   - Ensure `app.seed.enabled` remains false in runtime configs.
   - Remove any leftover temporary JSON test files (e.g., `tmp_login.json`) if they were only for local testing.
   - Files to check: `src/modules/crm/leads/page.tsx`, `backend/services/crm-service/src/main/java/com/fawnix/crm/common/config/DataSeeder.java`, `backend/services/identity-service/src/main/java/com/fawnix/identity/common/config/DataSeeder.java`, `backend/services/*/src/main/resources/application.yml`.

2. Implement Lead Import (CSV + XLSX) in CRM service
   - Add dependencies for CSV + XLSX parsing in `backend/services/crm-service/pom.xml` (e.g., Apache Commons CSV + Apache POI).
   - Add `LeadImportService` with validation/normalization and duplicate handling rules.
   - Add endpoint `POST /api/leads/import` (multipart form) accepting `.csv` and `.xlsx`.
   - Return: total rows, imported count, updated count, skipped count, row-level errors.
   - (Optional) Add audit tables `lead_import_batches` and `lead_import_errors` to store import history.
   - Files: `backend/services/crm-service/src/main/java/com/fawnix/crm/leads/controller/LeadController.java`, `backend/services/crm-service/src/main/java/com/fawnix/crm/leads/service/LeadImportService.java`, `backend/services/crm-service/src/main/resources/db/migration/V*_create_lead_imports.sql`.

3. Add Lead Import UI
   - Add “Import Leads” button + modal in leads page.
   - Allow file upload; show expected headers; show summary + row error list.
   - Files: `src/modules/crm/leads/page.tsx`, `src/modules/crm/leads/api.ts`, `src/modules/crm/leads/hooks.ts`, `src/modules/crm/leads/types.ts`.

4. Add Sales Visit + Demo Scheduling (backend)
   - New table `lead_schedules` (lead_id, type, status, scheduled_at, location, mode, notes, assigned_to_user_id/name, created/updated fields).
   - CRUD endpoints scoped to lead: list, create, update, update status.
   - Log lead activity when schedules are created/updated.
   - Files: `backend/services/crm-service/src/main/resources/db/migration/V*_create_lead_schedules.sql`, `backend/services/crm-service/src/main/java/com/fawnix/crm/leads/...` (entity, repository, service, controller, DTOs).

5. Add Scheduling UI in Lead Detail
   - Display upcoming and past schedules.
   - Form to create schedule; actions to mark complete/cancel.
   - Files: `src/modules/crm/leads/page.tsx`, `src/modules/crm/leads/types.ts`, `src/modules/crm/leads/api.ts`.

**Testing / Verification**
- API: `POST /api/leads/import` with CSV and XLSX files; verify summary + created/updated leads in DB.
- UI: import modal, error handling, list refresh.
- Schedules: create/update schedule via UI, verify DB row and activity log.
- Permissions: confirm sales reps can only interact with assigned leads.

**Notes**
- All new data will be persisted in Postgres; no JSON files will be used for runtime data.
- API is the source of truth; UI should not fall back to in-memory mocks.
